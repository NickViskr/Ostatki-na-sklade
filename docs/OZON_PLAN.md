## Реестр принятых решений (менять только по решению пользователя)
| Решение | Выбор |
|---|---|
| Дубли ручных и авто-заявок | Сопоставление по «отпечатку»: артикулы + количества + дата отгрузки (± окно) + назначение Ozon. Уверенное совпадение — автопривязка PostingID без создания дубля; спорное — на ручную сверку. С п.12 автопривязка уверенного дубля выполняется автоматически в момент обнаружения (клиент, окно уведомлений) |
| Доп. услуги | Только «сдаточные» (забор, упаковка, тариф по дате) считаются один раз при сдаче и входят в себестоимость. Хранение — отдельная метрика, в себестоимость поставки не входит |
| Момент ухода остатка | Статус Ozon ACCEPTED_AT_SUPPLY_WAREHOUSE. Заявки до отгрузки (DATA_FILLING, READY_TO_SUPPLY) оформить нельзя; проверка — хелпер isStockDeparted («принято на точке отгрузки или дальше»), т.к. опрос 2 раза в сутки может проскочить статус. Отказ, отмена и просрочка отгрузкой не считаются |
| Услуги в автопилоте | Автопилот обнаруживает отгрузку → всплывающее окно с префиллом позиций и сдаточных услуг → без утверждения пользователем расход не создаётся |
| Опрос Ozon | 2 раза в сутки, 05:00 и 17:00 МСК: триггеры Apps Script (Europe/Moscow) → прокси Cloud Run (https://service-415081166309.us-west1.run.app/) → Ozon Seller API → лист «Внешние отгрузки». Будильник — именно триггер Apps Script (Cloud Run засыпает). Плюс ручной запуск из Настроек |
| Недостача при приёмке | Остатки склада НЕ меняет. Стоимость разницы «отгружено − принято» перераспределяется на себестоимость товаров поставки. Каждая корректировка оставляет след в Истории |
| Пересорт | Признак: «заявлено X, принято меньше» + «заявлено 0, принято N». Проводится только после подтверждения пользователем. Ошибочный товар: списывается со склада + включается в состав поставки + услуги перераспределяются. Не уехавший: возвращается со своей базовой себестоимостью. Зеркалится во всех 6 путях учёта |
| Тестовая БД | Полная копия боевой; листы/колонки создаются идемпотентно (getOrCreateSheet); «Пользователи» чистится от паролей |
| Режим разработки | Тумблер только у админа, действует только в его браузере; запросы идут в тестовую БД; красная рамка + плашка «РЕЖИМ РАЗРАБОТКИ — ТЕСТОВАЯ БД» |
| Правило внедрения | Пункты 7–18 — только в режиме разработки на тестовой БД. Боевая база не участвует до п.20. Операции с остатками (пп. 12, 16) включаются только после просмотра «на показе» |
