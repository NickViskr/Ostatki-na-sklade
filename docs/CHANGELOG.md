# История изменений

## [2026-07-03]

### Добавлено
- Поля "Ozon Client-Id" и "Ozon Api-Key" (типа password) на вкладке "Настройки" (`SettingsTab`), которые сохраняются в глобальные параметры Google Таблиц через экшен `saveGlobalSettings` и доступны для изменения исключительно администраторам.
- Состояние `externalShipments: ExternalShipment[]` и экшены `checkOzonShipments()` и `markExternalShipment()` в `useWarehouseStore.ts`.
- Кнопка "Проверить отгрузки Ozon" на вкладке "Загрузка накладных" (`UploadTab`) для запуска проверки новых FBO отправлений.
- Панель "Обнаружены новые отгрузки Ozon" со списком новых поставок (ID, дата, количество штук) и действиями "Оформить" и "Игнорировать" для каждого заказа.
- Интеллектуальный механизм маппинга товаров в отгрузке: для каждого товара ищется SKU по штрихкоду (`barcode == ozonBarcode`) из списка `skus`. В случае отсутствия совпадения, сопоставление выполняется по артикулу Ozon (`offerId == SKU`). Если SKU не найден, элементу присваивается статус `'unknown'`.
- Интеграция с `ConfirmModal`: при нажатии "Оформить" данные передаются в `parsedItems`, статус операции выставляется в `Расход` со складом назначения `Ozon`, а после успешной записи транзакции в Google Таблицу (`commitTransaction` возвращает true) автоматически запускается смена статуса отгрузки в листе "Внешние отгрузки" на `'processed'`.
- Поддержка двух типов комплектов в системе: `legacy` (списание собственного остатка + компонентов) и `virtual` (списание только компонентов).
- Переключатель "Тип комплекта" в модальном окне редактирования комплекта (`KitModal`).
- Отображение расчетного количества сборки для виртуальных комплектов на вкладке «Дашборд» с пометкой «сборка».
- Разовая миграционная функция `migrateBowlKitsToVirtual()` в `Code.gs` для безопасного переноса `BowlGrayMini_01` и `BowlBlueMini_01` на виртуальную схему со списанием остатков на новые компоненты («Миска серая» и «Миска бирюзовая»).
- Колонка `kitType` в листе «Комплекты» в Google Таблицах.
- Новая разовая миграционная функция `migrateComponentWriteOffCosts()` для ручного обновления исторических данных транзакций в листе «Транзакции», приводящая поле «Себестоимость списания» у строк-компонентов в соответствие с полем «Сумма».
- Поддержка поля «КОР/ПАЛ» (boxesPerPallet) для SKU: 6-я колонка в листе SKU в Google Таблицах, позволяющая задавать количество коробок на одном паллете.
- Поле ввода «Коробок на паллете» в модальном окне добавления/редактирования SKU (`SkuModal`) с адаптивной трехколоночной сеткой.
- Отображение количества коробок («X кор.») для каждой товарной позиции в таблице подтверждения отгрузки в `ConfirmModal` (только для операций «Расход»).
- Суммарный информационный блок под таблицей в `ConfirmModal` (для операций «Расход») с отображением итогового количества коробок (`totalBoxes`) и итогового количества паллет (`totalPallets`).
- Умное однократное автозаполнение (предзаполнение) количества дополнительных услуг: если название активной услуги содержит «короб», её количество автоматически устанавливается равным `totalBoxes`, если содержит «паллет» — `totalPallets`.
- Поддержка расчета стоимости хранения товаров на складе: добавлена колонка «Литраж (л)» (volumeLiters) в лист SKU и глобальный параметр стоимости хранения за 1 литр в сутки `storageRatePerLiterDay` (props `global_storageRate`).
- Поля `ozonClientId` и `ozonApiKey` в глобальных настройках (`getGlobalSettings` и `saveGlobalSettings`) с доступом только для администраторов.
- Защищенный по сигнатуре (`verifyServerSignature`) серверный action `'getOzonKeys'`, возвращающий API-ключи Ozon.
- Новый лист Google Таблиц «Внешние отгрузки» для накопления заказов с Ozon Seller API с колонками `PostingID`, `Дата обнаружения`, `Дата отгрузки`, `Статус` ('new', 'processed', 'ignored'), `ПозицииJSON`, `TransGroupInfo`.
- Три новых action-обработчика для работы с внешними отгрузками для авторизованных пользователей: `'saveExternalShipments'` (добавление новых уникальных отправлений), `'getExternalShipments'` (выборка всех записей в виде JSON-объектов) и `'updateExternalShipmentStatus'` (изменение статуса отправления).
- Новый эндпоинт POST `/api/ozon/check` в `server.ts` для запуска синхронизации заказов FBO Ozon за 30 дней со встроенной валидацией сессий и автоматическим пробросом в Google Sheets.
- Механизм `fetchOzonKeys` в `server.ts` для безопасного извлечения API-ключей Ozon из GAS с использованием HMAC SHA256 подписи и кэшированием на 1 час.
- Инвалидация кэша ключей Ozon в `server.ts` при сохранении глобальных настроек (`saveGlobalSettings`).
- Поддержка интеллектуальных фалбеков по версиям (v2 -> v1) для Ozon API методов получения списка поставок (`supply-order/list`), деталей (`supply-order/get`) и состава (`supply-order/bundle`) для отказоустойчивости при deprecation и 404 ошибках.
- Поле ввода «Литраж, л» (шаг 0.01, необязательное) в модальном окне добавления/редактирования SKU (`SkuModal`).
- Блок «Хранение товаров» на вкладке «Справочник» (`DirectoryTab`) для настройки тарифа хранения за 1 л/сутки (редактирование доступно только системным администраторам, сохранение через `saveGlobalSettings`, загрузка через `getGlobalSettings`).
- Новая карточка аналитики на дашборде «Стоимость хранения» с отображением расходов в сутки и прогнозируемых затрат за месяц (сутки x 30).
- Новая колонка «Хранение ₽/сут» в таблице остатков на дашборде с возможностью динамической сортировки по стоимости хранения остатка.

### Изменено
- Логика проведения транзакций (`commitTransaction`): для типа 'virtual' при Расходе себестоимость главной строки сбрасывается в 0, а собственный складской остаток артикула комплекта не изменяется (списываются только компоненты).
- Логика удаления транзакций (`deleteTransaction`): при откате главной строки Расхода виртуального комплекта (с нулевой себестоимостью и заполненным `groupId`), собственный остаток артикула не возвращается на склад, возвращаются только компоненты.
- Валидация остатков перед отгрузкой: для `legacy` проверяется собственный остаток комплекта и компонентов, для `virtual` — только компонентов.
- Исправлена формула расчета `nCap` при каскадном удалении транзакции комплекта: теперь прибавляется прямая сумма себестоимости списания из соответствующей колонки вместо умножения количества на цену за единицу, так как в `Себестоимость списания` теперь пишется итоговая себестоимость (сумма) списания.

### Исправлено
- Исправлена валидация наличия компонентов при проведении отгрузок ('Расход') в функции `commitTransaction` (`Code.gs`): теперь потребность компонентов предварительно суммируется в ассоциативный массив `componentDemand` по всем списываемым комплектам в одной транзакции. Это предотвращает ложное прохождение валидации, когда комплекты в отгрузке по отдельности укладываются в остатки, но суммарно уводят компоненты в дефицит.
- Исправлена проверка остатков для виртуальных комплектов (type === 'virtual') во время проведения отгрузки и валидации распознанных файлов в `useWarehouseStore.ts`. Ранее для них проверялся собственный физический остаток на складе (всегда равный 0), что блокировало отгрузку. Теперь остаток виртуального комплекта рассчитывается динамически по минимальной обеспеченности его компонентами на основе новой функции `getEffectiveAvailability`, добавлена агрегированная проверка нехватки компонентов при отгрузке, а также отключена ложная ошибка «Артикул не найден».
- Исправлен поиск листа транзакций в функции `migrateComponentWriteOffCosts` (`Code.gs`): жесткий вызов `ss.getSheetByName('Транзакции')` заменен на вызов `getTransactionSheet(ss)`, что предотвращает падение миграции с ошибкой «Лист :Транзакции не найден» на таблицах, где лист назван «История».
- Исправлено вычисление `shipmentTotalQty` в функции `commitTransaction` (`Code.gs`): теперь при подсчете общего количества отгрузки отсекаются позиции, имеющие статус, отличный от `'ok'` (например, аннулированные или неизвестные SKU). Это обеспечивает верное пропорциональное распределение накладных/транспортных расходов на фактически проведенные в системе товары.
- Исправлен баг в функции `setupDatabase` (`Code.gs`), когда при пересборке листа SKU терялась колонка «Литраж (л)», так как expectedHeaders содержал только 6 колонок, а лист полностью очищался. Теперь expectedHeaders содержит 7 колонок, индекс литража определяется динамически, значение сохраняется при миграции строк, а размер диапазона корректно увеличен до 7.
- Исправлен баг в `deleteTransaction` (Code.gs), где использовались жесткие индексы `13` (groupId) и `14` (isComponent). Теперь позиции колонок определяются динамически на основе первой строки заголовков. Это восстановило корректный откат транзакций списания комплектов и компонентов.
- Исправлен баг в `commitTransaction` (Code.gs), из-за которого в поле `Себестоимость списания` для строк-компонентов записывалось значение за единицу (`compAvg`) вместо общей себестоимости (`compTotal`). Это приводило к некорректной работе функции полного пересчета остатков и восстановления.
- Исправлена ошибка `TypeError: Assignment to constant variable` в функции `setupDatabase` (Code.gs). Переменная `transSheet` переведена на `let` для возможности корректного переопределения при инициализации пустой базы данных.
- Устранена уязвимость в `/api/parse-invoice` (server.ts): добавлена обязательная проверка сессии пользователя (`sessionToken`) с запросом к Google Apps Script для проверки сессий, отсутствующих в локальном in-memory кэше.
- Устранена уязвимость в `/api/gas` (server.ts): кэшированные ответы из `gasCache` теперь отдаются только при наличии валидного и закэшированного `sessionToken`, чтобы исключить обход аутентификации.
- Обновлен клиентский вызов `parseInvoiceWithGemini` (src/lib/gemini.ts, src/store/useWarehouseStore.ts) для корректной передачи `sessionToken` первым параметром.
- Унифицирована проверка прав администратора в `Code.gs` с добавлением вспомогательной функции `isAdminRole`.
- В `assertAdmin` удалена проверка по имени пользователя, осталась только проверка роли через `isAdminRole`.
- В `getGlobalSettings` заменено жесткое сравнение роли на вызов `isAdminRole`.
- Реализована детальная диагностика ошибок распознавания накладных: эндпоинт `/api/parse-invoice` теперь передает детальный объект ошибки (`details`) со свойствами `stage`, `model`, `httpStatus` и `rawError`. На клиенте эти детали обрабатываются, выводя информативные `toast.error` (об отсутствии ключа, ошибках модели 404, лимитах 429 и невалидном JSON от модели).
- Синхронизирована дефолтная модель Gemini: устаревшая модель по умолчанию `gemini-1.5-flash` и `gemini-3-flash-preview` заменена на актуальную `gemini-flash-latest` в `server.ts`, `Code.gs` (в `getGlobalSettings`) и в дефолтном значении параметра `parseInvoiceWithGemini` в `src/lib/gemini.ts`.
- Добавлена кнопка очистки загруженных накладных на вкладке «Загрузка»: в `useUIStore` внедрен метод `clearRecognitionData` для сброса истории, сырого текста, файлов и результатов парсинга. На вкладке добавлена кнопка «Очистить накладные» с предупреждающим подтверждением через `askConfirmation` и последующим алертом об успешной очистке.
- Реализована история тарифов услуг по датам в `Code.gs`: добавлен лист «Тарифы услуг» (ServiceID, Стоимость, ДействуетС), поддерживаемый в `setupDatabase` и `getOrCreateSheet`. Реализованы функции `getServiceRates`, `getServiceCostAt(serviceId, dateStr, ratesArr, services)` и экшены `addServiceRate` (для администраторов) и `getServiceRates` (для всех авторизованных). Функция `getServices` теперь возвращает дополнительное поле `currentCost` на сегодняшнюю дату.
- Подключена история тарифов услуг по датам на фронтенде: в `src/types.ts` добавлен интерфейс `ServiceRate`, в `useWarehouseStore` интегрирована загрузка тарифов в `fetchStock` через `getServiceRates` и добавлен экшен `handleAddServiceRate` для вызова GAS `addServiceRate`.
- На вкладке «Справочник» (`DirectoryTab`) для каждой услуги добавлена кнопка «Тарифы», раскрывающая панель со списком тарифов (стоимость + дата начала действия, отсортированные по убыванию даты) и формой добавления нового тарифа (доступной только системным администраторам).
- В модальном окне подтверждения (`ConfirmModal`) расчеты себестоимости отгрузки и вывод стоимости услуг полностью переписаны на динамический выбор тарифа через функцию `getServiceCostAt(serviceId, deliveryDate)` в зависимости от выбранной пользователем даты поставки на маркетплейс, с автоматическим пересчетом и обновлением цен в реальном времени при изменении даты.
