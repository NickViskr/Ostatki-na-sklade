# Паспорт проекта «Интеграция с Ozon» (автопилот)

Статус-легенда: ⬜ не начат / 🟨 в работе / ✅ готов (+дата)

## Как работать с этим файлом
- Один пункт плана = один отдельный чат с архитектором (Claude).
- Новый чат начинается фразой: «Работаем по docs/OZON_PLAN.md из репозитория NickViskr/Ostatki-na-sklade, пункт N. Прочитай план и код».
- После завершения каждого пункта Gemini обновляет в этом файле статус пункта и раздел «Заметки для следующего чата».
- Этот файл — единственный источник правды по интеграции. При расхождении с чатами верить этому файлу.
- После завершения пункта архитектор (Claude) обязан: (а) выдать промт на отметку пункта ✅ с датой в этом файле, (б) напомнить пользователю закрыть текущий чат и открыть новый чат для следующего пункта стандартной фразой из этого файла.
- В начале каждого нового чата архитектор сверяет статусы пунктов в этом файле с фактическим кодом на GitHub; если пункт сделан, но не отмечен — сначала выдаётся промт на отметку, потом начинается работа.

## Реестр принятых решений (менять только по решению пользователя)
| Решение | Выбор |
|---|---|
| Дубли ручных и авто-заявок | Сопоставление по «отпечатку»: артикулы + количества + дата отгрузки (± окно) + назначение Ozon. Уверенное совпадение — автопривязка PostingID к ручной заявке без создания дубля; спорное — парковка на ручную сверку. Автопилот никогда не двигает остаток вслепую |
| Доп. услуги | Только «сдаточные» (забор груза, упаковка, тариф по дате) считаются ОДИН раз при сдаче поставки и входят в себестоимость. Хранение — отдельная метрика дашборда, в себестоимость поставки не входит |
| Момент ухода остатка со склада | Статус Ozon «принято на точке отгрузки» (ACCEPTED_AT_SUPPLY_WAREHOUSE). Позже данные поставки корректируются по приёмке на конечных складах (их может быть несколько) |
| Услуги в автопилоте | Автопилот обнаруживает поставку → при входе пользователя всплывающее окно с префиллом позиций и полем сдаточных услуг → пользователь проверяет/меняет услуги и утверждает. Без утверждения расход не создаётся |
| Опрос Ozon | Автоматически 4 раза в сутки: триггер по времени в Apps Script → прокси (Cloud Run: https://service-415081166309.us-west1.run.app/) → Ozon Seller API → запись в лист «Внешние отгрузки» |
| Недостача при приёмке | Остатки склада НЕ меняет (товар физически уехал весь). Стоимость разницы «отгружено − принято» перераспределяется на себестоимость товаров этой поставки. Каждая корректировка оставляет след в Истории |
| Пересорт | Признак в акте приёмки: «заявлено X, принято меньше» + «заявлено 0, принято N». Проводится ТОЛЬКО после подтверждения пользователем. Ошибочно отправленный товар: списывается с остатков склада + включается в состав поставки + сдаточные услуги перераспределяются на новый состав. Заявленный, но не уехавший товар: возвращается на склад со своей базовой себестоимостью. Операция зеркалится во всех 6 путях учёта |
| Тестовая БД | Полная копия боевой Google-таблицы «БД Склад»; все листы и колонки создаются автоматически (идемпотентная инициализация getOrCreateSheet); лист «Пользователи» при копировании чистится от паролей |
| Режим разработки | Тумблер только у админа, действует ТОЛЬКО в браузере включившего (боевых пользователей не затрагивает); все запросы в этом режиме идут в тестовую БД; красная рамка по периметру приложения + плашка «РЕЖИМ РАЗРАБОТКИ — ТЕСТОВАЯ БД» |
| Правило внедрения | Пункты 7–18 разрабатываются только в режиме разработки на тестовой БД. Боевая база не участвует до пункта 20. Операции, двигающие остатки (пп. 12, 16), включаются только после просмотра «на показе» |
